# Quest Board Implementation Plan

> **For Hermes / Claude Code:** Use this as the implementation brief for a new world feature. Keep rewards economy/cosmetic only. Do **not** add stat buffs, relic hooks, skill trees, or progression multipliers.

**Goal:** Add a physical `Quest Board` block that lets players browse and complete lightweight server quests without competing with the separate relic/achievement/buffs mod.

**Architecture:** Implement the Quest Board as a world block + menu/screen + persistent quest data store, similar in spirit to the existing Bounty Board. Reuse the existing wallet/economy flow and borrow simple turn-in logic patterns from `QuestGiverEntity`, but keep the board system data-driven and independent from NPCs.

**Tech Stack:** Fabric mod (current repo conventions), Java, existing menu/screen/block registration patterns in this repo, `WalletData` for rewards, saved data pattern similar to `BountyData`.

---

## Product intent

The Quest Board should feel like a shared town-square feature:
- a **placeable block** in spawn/towns
- gives players **daily / repeatable / community** tasks
- works alongside NPCs instead of replacing them
- rewards are **coins and/or cosmetics only**
- no buffs, no relic synergy, no power creep

This is intentionally a **server feature**, not a character progression system.

---

## Scope for V1

Build only these quest types in V1:

1. **Fetch quests**
   - Example: deliver 32 oak logs
   - Complete by checking/removing matching items from inventory

2. **Light exploration tasks**
   - Example: visit a biome / travel X blocks / visit a coordinate region
   - Keep implementation minimal; prefer one simple exploration condition in V1

3. **Community goals**
   - Example: server contributes 2,000 cobblestone together
   - Shared progress, one reward claim per player when completed

4. **Server board rotation**
   - A fixed list of active quests on the board
   - Manual refresh command for admins in V1; automatic daily rotation optional if cheap

Out of scope for V1:
- branching dialogue
- reputation systems
- buffs/perks/stats
- relic unlocks
- skill trees
- complex chains
- per-faction progression
- remote UI commands as the main access path

## V1 functional design

### Board behavior
When a player opens the Quest Board:
- show up to **5 active quests**
- each quest shows:
  - title
  - short description
  - reward
  - progress
  - status (`available`, `ready to turn in`, `completed today`, `claimed`)
- player can cycle through quests
- player can click:
  - `Track/Inspect`
  - `Turn In` for fetch quests
  - `Claim Reward` for community-goal completion
  - `Refresh` (client refresh only)

### Quest categories
#### 1. Fetch
- target item + required amount
- uses inventory scan/removal similar to `QuestGiverEntity`
- reward: wallet coins

#### 2. Exploration
Pick only one cheap implementation for V1:
- **Option A (recommended):** visit a biome
- **Option B:** travel total distance
- **Option C:** enter a configured region

Recommendation: do **biome visit** or **distance traveled**, whichever is easiest in this codebase.

#### 3. Community goal
- one server-wide shared progress counter
- players can contribute items from inventory
- once goal is met:
  - quest marked complete globally
  - each player can claim reward once

### Rotation
V1 should support:
- a static configured active quest list on startup OR
- generated defaults in code
- admin command to reroll/refresh active board quests

Daily automation is optional and should only be added if it is small and safe.

## Data model proposal

### `QuestType`
Use an enum:
- `FETCH`
- `EXPLORE_BIOME` or `TRAVEL_DISTANCE`
- `COMMUNITY_FETCH`

### `QuestDefinition`
Represents the quest template.

Suggested fields:
- `String id`
- `String title`
- `String description`
- `QuestType type`
- `long reward`
- `boolean repeatable`
- `String resetGroup` (e.g. `daily`, `weekly`, `none`)
- `ItemStack requiredItem` or item id + amount for fetch types
- biome id / distance target / community target depending on type
- `boolean cosmeticReward` optional later, but V1 can stay economy-only

### `QuestProgress`
Per-player progress for personal quests.

Suggested fields:
- `UUID playerId`
- `String questId`
- `int amountProgress`
- `boolean completed`
- `boolean claimed`
- `long lastCompletedDay`

### `QuestBoardData`
Persistent world save data.

Suggested contents:
- active quest definitions or active quest ids
- map of player progress
- map of community progress
- map of community reward claims
- last rotation day / board seed

Important: keep this independent from NPC entity save logic.

---

## UX / design rules

- The board should feel like a **public notice board**, not a character menu.
- Quest text should be short and readable.
- Rewards must stay in:
  - money
  - vanity / cosmetic tokens (future)
- Avoid anything that says:
  - `+damage`
  - `+speed`
  - `+health`
  - `buff`
  - `perk`
- NPCs can later reference the board, but the board must work **standalone**.

- There is a **placeable Quest Board block**.

- At least **3 quest categories** exist in V1:
  - fetch
  - one exploration type
  - community goal

---

## Recommended implementation shortcuts

To keep this fast and low-risk:
- Start with **hardcoded default quests** in Java instead of full JSON config.
- Use a **simple menu/screen** instead of a fancy paginated UI.
- Keep text short to avoid overcomplicated sync.
- Implement **community fetch** rather than abstract “community goal engine”.
- Choose **one** exploration type only.

---

## Risks / tradeoffs

1. **String sync in menus can get awkward**
   - Mitigation: keep titles/descriptions short in V1.

2. **Exploration event tracking may be annoying depending on hooks**
   - Mitigation: switch to distance traveled if biome visit is costly.

3. **Community claims can double-pay if claim state is sloppy**
   - Mitigation: explicit per-player claimed set in saved data.

4. **Feature creep toward progression system**
   - Mitigation: reject buffs/relic hooks in V1.
